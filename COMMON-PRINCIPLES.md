# 공통 작업 원칙

> 이 문서는 책 『나의 25년을 AI에게 가르치다』의 참고 자료입니다.
> 본문 6장 "반복되는 개발 작업도 자동화했다"와 부록 G에서 다룬, 스킬을 개발할 때 매 버전마다 따랐던 작업 원칙입니다.
>
> 이 원칙들은 처음에는 사람이 읽고 따르는 체크리스트 문서였지만, 버전이 올라가면서 상당 부분이 코드로 자동화되었습니다. 아래는 그 내용을 외부 독자가 읽을 수 있도록 구현 세부를 덜어내고 원칙 중심으로 정리한 것입니다. 파일명·스크립트명은 이 스킬의 실제 예시이며, 당신의 분야에서는 다른 이름이 될 것입니다.

---

## 1. 핵심 원칙 다섯 가지

### 원칙 1 — 검증 없는 수정은 부채다

코드를 손댄 뒤에는 예외 없이 테스트를 돌린다. 한 줄만 고쳤더라도 전체를 다시 확인한다. 검증을 미루면 나중에 더 큰 비용을 치른다.

> 책 7장 원리 1에 해당.

### 원칙 2 — 단일 진실 원천을 지킨다

버전 번호처럼 여러 곳에 나타나는 정보는 한 곳(`VERSION` 파일 등)을 원천으로 정하고, 나머지는 모두 그것과 일치시킨다. 어긋나면 배포 사고로 간주한다. 정보가 여러 곳에 흩어지면 반드시 어긋난다.

> 책 7장 원리 5에 해당.

### 원칙 3 — 조용한 오류를 용납하지 않는다

오류를 소리 없이 삼키지 않는다. 무언가 부족하거나 잘못됐으면, 무슨 작업에 무엇이 왜 필요한지를 구체적으로 드러낸다. 겉보기엔 멀쩡하지만 속으로 틀린 결과가 가장 위험하다.

> 책 6장·7장에서 거듭 강조한 경계심.

### 원칙 4 — 도메인 판단을 코드 구조보다 우선한다

구현의 편의를 위해 도메인의 진실을 굽히지 않는다. 코드를 깔끔하게 만드는 일과, 분야의 규칙을 정확히 지키는 일이 충돌하면 후자를 따른다.

> 책 7장 원리 6에 해당.

### 원칙 5 — 완료 조건을 코드로 정의한다

"빠뜨린 게 없는지"를 사람의 기억에 맡기지 않는다. 배포할 것이 규칙대로 들어갔는지, 버전이 맞는지, 파일 수가 한도 안인지를 코드가 직접 확인하게 한다. 사람은 빠뜨리지만 코드는 빠뜨리지 않는다.

## 2. 스킬 설계 원칙

*(v1.3.6부터 적용)*

- **SKILL.md는 라우팅 로직과 출력 정책만 담는다.** 세부 규칙은 `references/` 폴더로 분리한다.
- **요청은 "사용자 프롬프트 + 첨부 파일 상태 → 워크플로 선택 → 산출물 결정" 순서로 처리된다.** (예: `SKILL.md`의 라우팅 → 해당 워크플로 파일 → 산출물 생성)
- **자연어를 CLI 옵션보다 우선 인식한다.** "revision mark로", "검토해줘", "반영해줘" 같은 표현이 `--type`, `--mode` 같은 플래그보다 먼저 해석된다.
- **같은 파일을 첨부해도 프롬프트에 따라 다른 산출물이 나온다.** 프롬프트가 작업을 결정한다.
- **초안 단계에서는 MD + readiness check를 먼저, DOCX는 사용자 확인 후 생성한다.** 바로 최종 문서를 만들지 않는다.
- **약한 단독 표현("봐줘", "검토해줘")만으로는 신뢰 점수가 낮아진다.** 업무 맥락을 함께 제공할수록 정확한 워크플로로 연결된다.

## 3. 테스트 — 단계별 순서 준수

코드를 고칠 때마다 아래 순서를 지킨다. 순서를 건너뛰지 않는다.

**Step 1: 코드 수정 직후 — 즉시 실행**

단위 테스트와 통합 테스트를 돌린다. (예: `tests/unit/`, `tests/integration/` 폴더의 pytest) 이 단계가 통과하지 않으면 다음으로 가지 않는다.

**Step 2: E2E 품질 테스트 — 반드시 실행**

프롬프트 시나리오 전체를 실제로 실행하고 산출물이 의도대로 나오는지 검증한다. 실패하면 수정 후 재실행. (예: `tests/e2e/` + `run_tests.py`로 전체 시나리오 검증)

**Step 2.5: 프롬프트 라우팅 회귀 테스트 — 반드시 실행**

대표 프롬프트들이 여전히 올바른 워크플로로 연결되는지 확인한다. 새 프롬프트 항목을 추가했다면 이 테스트도 반드시 확장한다. (예: `tests/unit/test_prompts_routing.py`)

**Step 3: 전체 pytest + 결과 저장**

전체 테스트를 돌리고 결과를 보관한다. 다음 버전에서 회귀 여부를 확인하는 기준선이 된다.

**Step 3.5: 회귀 검증 인프라 실행** *(v1.8.5부터)*

골든 스냅샷(L1), 구조 불변식(L2), 라우팅 기준선(L3)을 한 번에 돌린다. (예: `run_all.py` → 0건 이상이면 배포 금지) 이 장치가 생기기 전에는 사람이 직접 확인했다.

**Step 4: 신규 기능 E2E 테스트 추가**

새 기능을 추가했으면 그에 맞는 테스트를 반드시 추가한다. 테스트 없는 기능은 다음 수정에서 조용히 망가질 수 있다.

## 4. 패키징 원칙

배포 ZIP을 만들 때 지키는 규칙이다. 실제 사례를 그대로 옮긴다.

**배포용 ZIP에 포함할 것:**
- `skill/` 폴더 내용만 포함
- `PROMPTS.md` / `PROMPTS.ko.md` 포함 필수 (스킬 등록에 활용)

**배포용 ZIP에 절대 포함하지 않을 것:**
- `tests/`, `eval/`, `logs/`, `dist/` — 개발 전용, 사용자 불필요
- `README.md`, `HISTORY.md` — 루트 전용 (배포본에 넣으면 파일 수 낭비)

**★ 파일 200개 제한** *(v2.0.0부터 필수)*

스킬 등록 시스템은 ZIP 내 파일 개수가 200개를 초과하면 등록을 거부한다. 패키징 전에 반드시 `skill/` 하위 파일 수(디렉토리 제외)를 세어 200개 이하인지 확인한다. 190개에 근접하면 경고하고 근본적 감축 방안을 검토한다.

이 제약이 v2.0.1에서 대규모 구조 정리를 이끌어낸 계기가 됐다. (책 부록 E 참조)

## 5. 수동 검증 체크리스트

자동화로 확인되지 않는 항목들을 배포 전에 사람이 직접 확인한다. 아래는 실제로 사용한 체크리스트다.

### 구조·버전

- [ ] `VERSION` 파일과 코드 내 `__version__` 이 일치하는가? (예: `build_itu_doc.py`의 `__version__`)
- [ ] `SKILL.md`의 버전 표기와 `VERSION`이 일치하는가?
- [ ] `PROMPTS.ko.md`와 `PROMPTS.md`의 내용이 동기화돼 있는가?
- [ ] `skill/` 하위 파일 수가 200개 이하인가?

### 테스트·회귀

- [ ] `run_all.py` (또는 동등한 회귀 검증 스크립트)가 0건으로 통과했는가?
- [ ] 새로 추가한 기능에 대한 E2E 테스트를 작성했는가?
- [ ] 새 PROMPTS 항목을 추가했다면 라우팅 회귀 테스트도 추가했는가?
- [ ] 이전 버전과 동일한 입력에서 동일한 산출물이 나오는가? (골든 스냅샷 비교)

### 배포 패키징

- [ ] 배포 ZIP에 `tests/`, `eval/`, `logs/`, `dist/`가 포함되지 않았는가?
- [ ] 배포 ZIP에 `README.md`, `HISTORY.md`가 포함되지 않았는가?
- [ ] 배포 ZIP 파일명이 규칙(`ITU-document-creator-<version>.zip`)을 따르는가?
- [ ] 전체 프로젝트 ZIP(작업 결과물 전달용)과 배포용 ZIP이 구분되어 있는가?

### 산출물 품질

- [ ] 대표 프롬프트 3~5개를 실제로 실행해서 산출물이 의도대로 나오는가?
- [ ] DOCX 산출물의 헤더·스타일이 ITU 양식을 따르는가?
- [ ] 조용한 오류(silent failure)가 없는가? 오류 시 구체적인 메시지가 출력되는가?
- [ ] 새 기능이 기존 기능을 망가뜨리지 않았는가?

### 문서·이력

- [ ] `HISTORY.ko.md`에 이번 버전 변경 내용을 기록했는가?
- [ ] 중요한 설계 결정은 `CONTRIBUTING.md`에 기록했는가?
- [ ] `PROMPTS.ko.md`에 새 항목을 추가했다면 그룹·번호·버전 표기가 맞는가?

## 당신의 분야로

무언가를 반복해서 만들고 고친다면, 당신에게도 이런 공통 작업 원칙이 필요할 수 있습니다. 위의 항목들을 그대로 쓸 수는 없겠지만, 그 구조는 옮길 수 있습니다.

- 코드를 고쳤을 때 반드시 거치는 확인 단계는 무엇인가
- 배포할 때 빠뜨리면 안 되는 것은 무엇인가
- 사람이 기억에 의존하는 대신 코드가 확인할 수 있는 것은 무엇인가

이 질문들에 대한 답을 적어 두는 것이 당신의 공통 작업 원칙의 시작입니다.
